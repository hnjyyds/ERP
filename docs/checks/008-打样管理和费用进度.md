## 本次实现范围

后端：

- `GET /api/v1/sample/requests`：打样单列表，支持编号/客户/产品/供应商/要求/明细模糊查询，支持状态和客户过滤。
- `GET /api/v1/sample/requests`：补充支持打样日期 `date_from`/`date_to` 范围过滤。
- `POST /api/v1/sample/requests`：新增打样单，并把客户打样要求转为内部打样单明细。
- `GET /api/v1/sample/requests/{request_id}`：打样单明细、内部打样单明细、进度记录和费用记录。
- `POST /api/v1/sample/requests/{request_id}/progress`：新增打样进度，并同步打样单状态。
- `POST /api/v1/sample/requests/{request_id}/fees`：登记打样费用。
- `POST /api/v1/sample/requests/{request_id}/fees/{fee_id}/payment-request`：发起打样费用付款申请，生成正式合作伙伴费用发票和费用付款申请，并回写财务发票号/付款申请号。
- `POST /api/v1/sample/requests/{request_id}/sample-record`：打样完成后转为样品登记，并自动写入收样数量事件和确认样/大货样跟单事件。
- `GET /api/v1/printing/sample-requests/{request_id}`：生成内部打样单 HTML 打印视图。
- SQLAlchemy async model：`SampleRequest`、`SampleRequestLine`、`SampleRequestProgress`、`SampleFee`。
- Repository 返回 typed row。
- Service 负责权限校验、状态校验、去向校验、状态流转、费用付款申请号生成和数据权限过滤。
- FastAPI 路由显式声明 `response_model=ApiResponse[...]`。
- 打样管理菜单权限：`sample:request:view`、`edit`、`view_all`、`fee:edit`。

前端：

- React 左侧菜单接入“打样管理”。
- React 打样管理页支持列表、搜索、状态筛选、客户筛选、新增打样单、进度更新、费用登记和付款申请入口。
- React 打样管理页补充打样日期范围筛选、内部打样单打印和“打样完成转样品登记”表单。
- 打样去向覆盖：内部打样、外发工厂。
- 打样状态覆盖：草稿、已发送、进行中、已完成、已取消。
- 业务主管可访问打样管理；财务角色无打样管理菜单且接口返回 `403`。
- 页面沿用 ERP 工作区布局，便于业务员和跟单人员持续查看打样进度。

## 端到端证据

- 截图：`artifacts/e2e-sample-requests.png`
- E2E 覆盖：
  - 登录业务主管账号。
  - 点击“打样管理”菜单。
  - 等待 `/api/v1/sample/requests` 返回。
  - 新增外发工厂打样单，并录入内部打样单明细。
  - 等待 `/api/v1/sample/requests` 201 和列表刷新。
  - 更新打样进度为“外发工厂/进行中”。
  - 等待 `/api/v1/sample/requests/{request_id}/progress` 201。
  - 登记打样费用。
  - 等待 `/api/v1/sample/requests/{request_id}/fees` 201。
  - 发起打样费用付款申请。
  - 等待 `/payment-request` 成功并显示 `SAMPLE-FEE-` 申请号。

## 可用性检查

- 打样列表、状态筛选、客户筛选和新增表单在同一工作区可见。
- 打样详情集中展示客户、产品、去向、状态、日期、供应商和要求。
- 进度记录和费用记录位于同一明细区，方便跟踪“打样进度 + 费用处理”。
- 费用表内可直接发起付款申请，减少跨页面跳转。
- 财务角色不会看到打样管理菜单，避免越权入口。
