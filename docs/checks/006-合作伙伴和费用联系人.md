## 本次实现范围

后端：

- `GET /api/v1/masterdata/partners`：合作伙伴列表，支持编号/中英文名称/国家/联系人模糊查询，支持伙伴类型过滤。
- `POST /api/v1/masterdata/partners`：新增合作伙伴，同时写入主联系人。
- `GET /api/v1/masterdata/partners/{partner_id}`：合作伙伴明细和联系人。
- `PUT /api/v1/masterdata/partners/{partner_id}`：编辑合作伙伴基础资料和伙伴类型。
- `POST /api/v1/masterdata/partners/{partner_id}/contacts`：追加联系人，可替换主联系人。
- `GET /api/v1/masterdata/partners/{partner_id}/fee-records`：费用记录入口，当前返回空列表，后续由费用申请、单证、付费管理接入。
- SQLAlchemy async model：`Partner`、`PartnerContact`。
- Repository 返回 typed row。
- Service 负责权限校验、伙伴类型校验、主联系人唯一性、数据权限过滤和响应组装。
- FastAPI 路由显式声明 `response_model=ApiResponse[...]`。
- 合作伙伴菜单权限：`masterdata:partner:view`、`edit`、`view_all`。

前端：

- React 左侧菜单显示“合作伙伴”。
- React 合作伙伴页支持列表、搜索、类型筛选、新增合作伙伴、编辑合作伙伴、联系人列表、追加联系人、费用记录入口。
- 伙伴类型覆盖：快件公司、货代公司、保险公司、运输公司。
- 业务主管可访问合作伙伴；财务角色无合作伙伴菜单且接口返回 `403`。
- 页面沿用 ERP 工作区布局，便于业务、单证和财务后续引用同一份伙伴资料。

## 可用性检查

- 合作伙伴列表、类型筛选和新增表单在同一工作区可见。
- 合作伙伴明细集中展示名称、类型、国家、状态、主联系人和费用记录数量。
- 联系人追加和合作伙伴编辑位于明细区，和新增合作伙伴表单职责分离。
- 类型值在前端显示为业务名称，减少直接暴露系统编码。
- 财务角色不会看到合作伙伴菜单，避免越权入口。
