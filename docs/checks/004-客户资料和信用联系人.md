## 本次实现范围

后端：

- `GET /api/v1/masterdata/customers`：客户列表，支持编号/中英文名称/国家/联系人模糊查询，支持国家和信用等级过滤。
- `POST /api/v1/masterdata/customers`：新增客户，同时写入主联系人和信用信息。
- `GET /api/v1/masterdata/customers/{customer_id}`：客户明细、联系人、信用信息。
- `PUT /api/v1/masterdata/customers/{customer_id}`：编辑客户基础资料和信用信息。
- `POST /api/v1/masterdata/customers/{customer_id}/contacts`：追加联系人，可替换主联系人。
- `GET /api/v1/masterdata/customers/{customer_id}/transactions`：客户交易记录入口，当前返回空列表，后续由报价/出口合同/出货/收款接入。
- SQLAlchemy async model：`Customer`、`CustomerContact`、`CustomerCreditProfile`。
- Repository 返回 typed row，不向 Service 暴露无约束 dict。
- Service 负责权限校验、主联系人唯一性、授信额度字段权限、数据权限过滤和响应组装。
- FastAPI 路由显式声明 `response_model=ApiResponse[...]`。
- 客户资料菜单权限：`masterdata:customer:view`、`edit`、`credit:view`、`credit:edit`、`view_all`。

前端：

- React 左侧菜单接入“客户资料”。
- React 客户资料页支持列表、搜索、国家/信用等级过滤、新增客户、编辑客户、联系人列表、追加联系人、信用信息、交易记录入口。
- 业务主管可访问客户资料；财务角色无客户资料菜单且接口返回 `403`。
- 页面沿用 ERP 工作区布局，表格和表单在桌面宽度可扫描。

## 可用性检查

- 客户列表、查询和新增表单在同一工作区首屏可见。
- 客户明细集中展示名称、国家、状态、信用、授信额度、主联系人和交易记录数量。
- 联系人追加和客户编辑位于明细区，和新增客户表单职责分离。
- 表格单元格、筛选条件和按钮文字无重叠。
- 财务角色不会看到客户资料菜单，避免越权入口。
